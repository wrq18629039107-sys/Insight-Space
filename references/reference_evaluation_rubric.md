# Reference Evaluation Rubric

This file defines how Insight-Space judges whether a reference is strong, usable, visual-only, pending, or not recommended.

## Core Rule

A reference is valuable only when it explains:

- what the current project can learn;
- what should not be copied;
- what conditions are required for transfer;
- where cost, construction, maintenance, operation, safety, climate, regulation, or cultural risks may appear.

Do not use “good looking” as a selection reason.

## Minimum Information Before Evaluation

Required:

- project type;
- use scenario;
- core goal.

Useful but not always required:

- location and climate;
- area, scale, or site condition;
- user group and operation goal;
- budget and construction constraints;
- delivery audience;
- expected output mode.

If missing information is not fatal, make assumptions and mark them. Do not stop the work.

## Reference Grades

| Grade | Meaning | Use |
|---|---|---|
| Strong reference | highly relevant, method value clear, source reliable, risk understandable or manageable | deep analysis and final case document |
| Usable reference | locally useful, but limited by scale, budget, scene, source, or transfer condition | medium document, supplementary card, or local strategy |
| Visual clue | useful for form, atmosphere, material, composition, detail, or image direction, but facts incomplete | card or visual clue section only |
| Pending verification | potentially useful but missing key source facts | source-link list or candidate supplement, not final formal case |
| Not recommended | source weak, relevance low, risk too high, or no clear value | reject or record rejection reason in deep mode |

## Candidate Pool And Selection

Search more than the final output count.

| Mode | Candidate Pool | Final Formal Case Output | Card / Supplement Use |
|---|---:|---:|---|
| Project search - deep | default 10-16 candidates; extend only if needed | one deep report with 3-5 project items | no default card visual output; candidate cards only if requested |
| Project search - medium | default 7-12 candidates | one medium report with 5-8 project items | no default card visual output; candidate cards only if requested |
| Project search - shallow | default 5-8 candidates or visual directions | one shallow report with 8 project items by default | no default card visual output; candidate cards only if requested |
| Visual search | default 5-8 visual clues | no formal case count required | card visuals only, plus source links |

Selection is not “most visually similar”. Choose cases that provide different kinds of value.

Do not fully score every rough candidate. Use quick rejection for obviously weak or repetitive material, then apply the full rubric only to serious candidates and final selected cases. The goal is high-quality selection, not a full audit trail for every search result.

## Hard Entry Conditions For Selected Formal Cases

A selected formal case should:

- exist as a real project or clearly identified reference;
- have source support for most key facts;
- relate to the user project by type, scene, problem, user group, goal, strategy, material, or operation;
- be chosen for more than style similarity;
- explain what can be learned;
- explain what should not be copied;
- include at least one risk note.

If it fails these conditions, downgrade it to visual clue, pending reference, backup, or reject it.

## Selection Scoring

For internal judgment, score from 1-5 per dimension and convert roughly to a 100-point decision if useful.

| Dimension | Weight | High Value Means |
|---|---:|---|
| relevance | 20% | project type, scene, scale, people, goal, or problem strongly matches |
| method value | 20% | can extract spatial, path, material, operation, or experience strategy |
| transferability | 15% | can become practical design actions in the current project |
| source reliability | 15% | source is official, professional, or cross-checkable |
| implementation fit | 15% | cost, construction, and maintenance are manageable or can be simplified |
| difference value | 5% | adds a different angle from other selected cases |
| risk insight | 10% | helps reveal what may go wrong |

Suggested grade:

- 85-100: strong reference;
- 70-84: usable reference;
- 50-69: visual clue or pending reference depending on source and risk;
- below 50: not recommended.

## Evaluation Dimensions

| Dimension | Judge This | Good Sign | Weak Sign |
|---|---|---|---|
| scene fit | is the site type close to the current project? | same or adjacent scene | only mood is similar |
| function fit | does it solve a similar problem? | clear relation to rest, display, guide, shade, stay, interaction, or photo | only a sculptural look |
| scale fit | can volume and human scale transfer? | scale can be directly used or decomposed | large high-budget project forced into small site |
| form value | does form serve behavior, sequence, structure, or memory? | form can become an actionable design logic | only photogenic shape |
| material and craft | are materials and details buildable? | replaceable material, clear detail, realistic craft | rare material or high customization |
| user behavior | does space support real use? | stay, walk, interact, photograph, rest, browse, or gather | visual object without use logic |
| communication value | is there a memorable or shareable point? | photo point tied to function or path | short-lived “internet famous” styling |
| transferability | can the strategy become a design action? | can map to plan, interface, path, lighting, node, operation | can only copy surface |
| copy risk | what cannot be copied? | risk can be simplified or avoided | risk is high and inseparable from the concept |

## Cost / Construction / Maintenance Risk

Use separate 1-3 scores rather than one vague “difficulty” label.

| Score | Cost | Construction | Maintenance |
|---|---|---|---|
| 1 low | conventional materials, standard components, limited customization | common process, ordinary team can build, forgiving details | durable, easy to clean, low maintenance |
| 2 medium | some custom furniture, special finish, or non-standard pieces | needs clear details, cross-discipline coordination, higher precision | periodic cleaning, lighting/furniture/plant maintenance |
| 3 high | heavy non-standard work, rare/imported materials, complex structure or MEP | complex structure, high-precision joints, special process, site constraints, approvals | frequent maintenance, high wear, hard cleaning, demanding planting or operation |

Composite:

- 3-4: low risk, still check scale/site/regulation;
- 5-7: medium risk, borrow parts and explain conditions;
- 8-9: high risk, do not copy directly; extract concept, space organization, material mood, or local detail.

If any single score is 3 and conflicts with the user’s constraints, mark the whole strategy high risk.

When budget or construction conditions are missing, write “temporarily judged under medium-budget assumptions” rather than pretending precision.

## Deep Case Analysis Fields

For a deep or selected formal case, include:

- project name and source-supported facts;
- why selected;
- what problem it solves;
- three most valuable reference points when possible;
- usable strategies for the current project;
- non-copy points;
- cost / construction / maintenance notes;
- current-project recommendation;
- one-sentence conclusion;
- source and cross-check links.

## Medium Case Fields

Medium case documents reduce depth but keep judgment:

- images;
- project name, type, designer/location/year when verifiable;
- source;
- 2-3 most valuable points;
- 2-3 current-project suggestions;
- concise non-copy notes;
- recommendation grade;
- simplified risk note only when important.

## Rejection And Downgrade Reasons

Downgrade or reject when:

- only images exist and project facts cannot be traced;
- source is Pinterest/Instagram/image search with no traceable original;
- the project is only stylistically similar;
- scale, budget, construction, or operation are seriously mismatched;
- several cases repeat the same design move;
- the case cannot answer “why is this selected?”;
- risk is much higher than value.

## Quality Red Lines

Do not:

- say every case is worth referencing;
- choose high-budget projects for low-budget requests without simplifying;
- omit non-copy points;
- omit source type;
- hide uncertainty;
- judge only visual style;
- ignore user behavior and operation.
